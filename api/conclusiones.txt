1. RESUMEN DE RESULTADOS
-------------------------
- Las 4 pruebas de APIs REST se ejecutaron exitosamente.
- Se validaron los endpoints de signup y login con diferentes escenarios.
- Tiempo promedio de ejecucion: ~10 segundos.
- Resultado: 4 tests passing, 0 failing.

2. HALLAZGOS TECNICOS
----------------------

2.1 Comportamiento de la API Signup
   - Endpoint: POST https://api.demoblaze.com/signup
   - Metodo: POST con Content-Type: application/json
   - Body requerido: {"username": "string", "password": "string"}
   
   Respuestas observadas:
   - Exito: Retorna string vacio "" con status 200
   - Error (usuario existente): Retorna {"errorMessage": "This user already exist."} con status 200
   
   Hallazgo importante: La API retorna status 200 incluso en caso de error,
   diferenciando el resultado por el contenido del body (string vacio vs objeto con errorMessage).

2.2 Comportamiento de la API Login
   - Endpoint: POST https://api.demoblaze.com/login
   - Metodo: POST con Content-Type: application/json
   - Body requerido: {"username": "string", "password": "string"}
   
   Respuestas observadas:
   - Exito: Retorna "Auth_token: [token_base64]" con status 200
   - Error (password incorrecto): Retorna {"errorMessage": "Wrong password."} con status 200
   
   Hallazgo importante: Similar al signup, la API retorna status 200 en todos los casos,
   usando el contenido del body para indicar exito o error.

2.3 Generacion de usernames unicos
   - Para evitar conflictos entre ejecuciones de pruebas, se generaron
     usernames unicos usando timestamp y numero aleatorio.
   - Formato: user_[timestamp]_[random_number]
   - Esto garantiza que cada ejecucion de pruebas sea independiente.

2.4 Tiempo de espera entre signup y login
   - Se observo que es necesario esperar ~1 segundo entre el signup
     y el login para que el usuario este disponible en el sistema.
   - Se utilizo cy.wait(1000) para garantizar la consistencia de las pruebas.

2.5 Uso de failOnStatusCode: false
   - Las APIs retornan status 200 incluso en casos de error.
   - Se configuro failOnStatusCode: false en cy.request() para evitar
     que Cypress falle automaticamente y poder validar las respuestas manualmente.

3. HALLAZGOS FUNCIONALES
-------------------------

3.1 Validacion de usuario duplicado
   - La API signup valida correctamente si un usuario ya existe.
   - El mensaje de error es claro: "This user already exist."
   - No se permite crear usuarios duplicados.

3.2 Validacion de credenciales en login
   - La API login valida correctamente username y password.
   - Si el password es incorrecto, retorna: "Wrong password."
   - Si las credenciales son correctas, genera un Auth_token en formato base64.

3.3 Formato del Auth_token
   - El token de autenticacion tiene el formato: "Auth_token: [base64]"
   - Ejemplo: "Auth_token: dXNlcl85MzE0NTcxNzgxMDQx"
   - El token parece ser una codificacion base64 del username con timestamp.

3.4 Ausencia de validacion de campos vacios
   - No se probaron casos con username o password vacios.
   - Seria recomendable agregar pruebas para validar el comportamiento
     con campos vacios, null, o tipos de datos incorrectos.

4. VENTAJAS DE CYPRESS PARA PRUEBAS DE APIs
--------------------------------------------
- cy.request() permite hacer peticiones HTTP directamente sin necesidad de UI.
- Las aserciones de Cypress (expect) son potentes y legibles.
- Los comandos personalizados (cy.signup(), cy.login()) facilitan la reutilizacion.
- El manejo de fixtures permite externalizar datos de prueba.
- La generacion de video y screenshots ayuda en la depuracion.
- No requiere configuracion adicional de herramientas como Postman o Insomnia.

5. LIMITACIONES Y MEJORAS POTENCIALES
---------------------------------------
- No se probaron casos de error con campos vacios o invalidos.
- No se validaron diferentes longitudes de username/password.
- No se probaron caracteres especiales en username/password.
- No se realizaron pruebas de carga o rendimiento.
- No se integraron las pruebas con CI/CD (GitHub Actions, Jenkins).
- Se podrian agregar reportes avanzados con cypress-mochawesome-reporter.
- No se validaron tiempos de respuesta (performance testing).
- No se probaron diferentes metodos HTTP (GET, PUT, DELETE) en los endpoints.

6. COMPARACION CON OTRAS HERRAMIENTAS
--------------------------------------
Cypress vs Postman:
- Cypress: Integrado con el framework de pruebas E2E, permite combinar pruebas UI y API.
- Postman: Especializado en APIs, tiene mejor UI para exploracion manual.

Cypress vs Karate:
- Cypress: JavaScript, mas flexible, mejor para equipos de desarrollo web.
- Karate: Sintaxis Gherkin, mas legible para no-programadores, mejor para BDD.

Cypress vs REST Assured:
- Cypress: JavaScript, unificado con pruebas E2E.
- REST Assured: Java, mas maduro para pruebas de APIs complejas.

7. CONCLUSION GENERAL
-----------------------
Cypress.io resulto ser una herramienta efectiva para pruebas de APIs REST,
especialmente cuando ya se utiliza para pruebas E2E de UI. La capacidad de
usar cy.request() permite validar servicios backend sin necesidad de herramientas
adicionales como Postman o Insomnia.

Las APIs de demoblaze.com (signup y login) funcionan correctamente y validan
adecuadamente los casos de negocio principales:
- Creacion de usuarios nuevos
- Prevencion de usuarios duplicados
- Autenticacion con credenciales correctas
- Rechazo de credenciales incorrectas

El enfoque de generar usernames unicos con timestamp garantiza que las pruebas
sean independientes y repetibles, evitando conflictos entre ejecuciones.

Para un proyecto real, se recomendarian pruebas adicionales:
- Validacion de campos vacios e invalidos
- Pruebas de seguridad (SQL injection, XSS)
- Pruebas de performance y carga
- Validacion de rate limiting
- Pruebas de diferentes metodos HTTP
- Validacion de headers y cookies
